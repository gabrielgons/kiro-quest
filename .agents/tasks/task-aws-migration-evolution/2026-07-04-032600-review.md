# AWS migration: S3/CloudFront hosting, Cognito auth, Lambda/DynamoDB backend, CI/CD

A full-stack AWS migration for the Kiro Quest Vue 3 quiz game. Replaces Cloudflare Workers hosting with S3 + CloudFront, adds Cognito-based Gmail SSO (custom PKCE OIDC client with zero external dependencies), a Lambda + API Gateway + DynamoDB backend for persisting user progress and rankings, four GitHub Actions workflows using OIDC, and CDK infrastructure-as-code across four stacks. The app remains fully functional without any AWS services configured — auth and API degrade gracefully to the existing localStorage path.

**Watch for:** CORS wildcard origin on the API Gateway means any origin can call authenticated endpoints with a stolen token (confirmed). The DynamoDB ranking sort key uses zero-padded score strings that sort descending only because `ScanIndexForward: false`, but the sort key includes timestamp — so ties break by *most recent last*, which is backwards from typical leaderboard semantics (confirmed). The `persistToCloud` call in the quiz store is fire-and-forget with no retry or user feedback on failure (confirmed). The GitHub OIDC role has a `cloudfront:*` distribution wildcard rather than scoping to the specific distribution (confirmed).

## High-level view

The CDK infrastructure is cleanly partitioned into four stacks (Frontend, Auth, Backend, GitHub OIDC) with an optional DNS stack that synthesizes only when domain parameters are provided. The Backend stack depends on the Auth stack's user pool at synth time, coupling their deployment order.

The auth implementation is a from-scratch OIDC/PKCE client using Web Crypto and fetch — no Amplify, no oidc-client-ts. This keeps bundle size tiny but means the implementation must handle every edge case that libraries normally absorb: token rotation during concurrent requests, popup/redirect error recovery, and cross-tab session consistency. The current implementation covers the happy path; the refresh token race condition is the main gap.

The backend is five Lambda functions behind an HTTP API with a Cognito JWT authorizer. DynamoDB uses single-table design with composite keys. The API has no rate limiting beyond API Gateway's default account-level throttle, and no input size validation beyond checking required fields.

The CI/CD pipeline uses GitHub OIDC throughout (no stored secrets), but the IAM role's permissions are broader than needed — particularly the CloudFront wildcard and the `cdk-*` role assumption pattern that covers all CDK roles in the account.

The frontend's dual-mode persistence (localStorage always, cloud when authenticated) means unauthenticated users experience zero behavioral change. The gap is the absence of a cloud-to-local restore path — an authenticated user on a new device sees no progress until `restoreFromCloud()` is added.

<details>
<summary>Issues (7)</summary>

1. **CORS wildcard origin** — API Gateway allows `*` origin. Restrict to the CloudFront domain (and localhost in dev) so that stolen tokens cannot be replayed from arbitrary origins.
2. **Ranking tie-break order** — Sort key `SCORE#00009#<timestamp>#<userId>` with `ScanIndexForward: false` returns highest scores first but within the same score, latest timestamp wins. Typical leaderboards rank first-to-achieve higher. Reverse the timestamp component or use `MAX_TS - timestamp`.
3. **Cloud persistence is fire-and-forget** — `persistToCloud` swallows errors silently. If the API fails repeatedly, the user has no indication their progress is not synced. Add a "sync failed" UI indicator or retry queue.
4. **GitHub OIDC role CloudFront scope** — The `CloudFrontInvalidation` policy allows `distribution/*`, covering all distributions in the account. Scope to the specific distribution once its ID is known (pass it from the Frontend stack output).
5. **No request body size validation** — Lambda handlers parse `JSON.parse(event.body)` without checking body length. A large payload (API Gateway's 10MB limit) could inflate DynamoDB item size beyond 400KB limit, causing a 500.
6. **Refresh token race condition** — `getAccessToken()` calls `refreshTokens()` without a mutex. Two concurrent API calls with an expired token will both attempt refresh; the second will likely fail because Cognito invalidates the previous refresh token on use.
7. **Google client secret in plaintext** — `SecretValue.unsafePlainText(googleClientSecret)` stores the Google OAuth client secret in the CloudFormation template in plaintext. Use `SecretValue.secretsManager()` or SSM SecureString reference instead.

## Custom PKCE/OIDC client and token lifecycle

The stored PKCE verifier and state are cleaned up in the `cleanup()` call even on failure paths. If a user refreshes the callback page after a transient network error, the flow cannot be retried — they must start over from login.

The `refreshTokens` function has a concurrency issue. If two components call `getAccessToken()` simultaneously when the token is expired, both send the same refresh token to Cognito. Cognito invalidates the token after the first successful exchange — the second call gets "invalid_grant", triggering `clearTokens()` and logging the user out unexpectedly. A `Promise` deduplication pattern (memoizing the in-flight refresh) would prevent this.

```typescript
// Current code in client.ts — no deduplication:
export async function getAccessToken(): Promise<string | null> {
  const tokens = getStoredTokens();
  if (!tokens) return null;
  if (isTokenExpired(tokens)) {
    const refreshed = await refreshTokens();
    return refreshed?.accessToken ?? null;
  }
  return tokens.accessToken;
}
```

Refresh tokens are stored in localStorage alongside access and id tokens. No CSP `script-src` directive is configured — the CloudFront security headers policy sets X-Frame-Options and HSTS but not Content-Security-Policy. XSS would grant access to the refresh token directly.

## DynamoDB single-table design and ranking sort order

The ranking sort key is constructed as:
```typescript
const paddedScore = String(body.correctCount).padStart(5, '0');
`SCORE#${paddedScore}#${completedAt}#${userId}`
```

With `ScanIndexForward: false`, this yields descending lexicographic order. The primary score dimension works correctly (`SCORE#00010` sorts above `SCORE#00009`). But within the same score, higher timestamps sort first (because `1700000002000` > `1700000001000` and we're reversing). If Alice scores 10 at T=100 and Bob scores 10 at T=200, Bob appears higher — rewarding lateness rather than being first to achieve the score.

The fix is to invert the timestamp: `MAX_TIMESTAMP - completedAt` (zero-padded), so earlier completions sort first within the same score tier.

The `getProfile` handler performs a read-modify-write cycle (GET, compute derived fields, PUT) without any conditional write. Two concurrent requests could overwrite each other's updates. Low risk for a quiz profile, but worth a conditional expression if the profile gains user-editable fields.

## CORS and API Gateway security posture

The HTTP API configures CORS with `allowOrigins: ['*']`. The access token lives in localStorage on the legitimate origin, so direct cross-origin theft requires XSS. However, wildcard CORS is a defense-in-depth gap — if the token leaks via any channel (browser extension, debug page, etc.), it can be used from any origin. Restricting to the CloudFront domain closes this.

There is no request throttling beyond API Gateway's account-level default (10,000 req/s). A single user spamming `saveProgress` could rack up DynamoDB WCU costs. Consider a per-route throttle in the HTTP API stage configuration.

## Cloud persistence and the localStorage fallback

```typescript
api.saveProgress(request).catch(() => {
  // Cloud sync failed - localStorage has the data as fallback
});
```

If the API is misconfigured or the token expires mid-session, every save silently fails. The user believes progress is synced but it's only local. On device switch, the progress is gone.

There is no cloud-to-local restore path. `progressTracker.restore()` only reads from localStorage. An authenticated user on a new device sees no progress until a `restoreFromCloud()` path is added that calls `api.getProgress()` and hydrates the store on login.

## GitHub Actions OIDC and IAM permissions

The `CloudFrontInvalidation` policy targets `distribution/*`:
```
resources: [`arn:aws:cloudfront::${this.account}:distribution/*`]
```
This allows invalidating any CloudFront distribution in the account. The S3 policy is correctly scoped to `kiro-quest-site-${this.account}` and CDK policies use `KiroQuest*` stack name prefixes.

The OIDC trust uses `repo:${props.repositoryName}:*`, covering all branches and events. The current workflows use `pull_request` (not `pull_request_target`), so fork PRs cannot assume the role. If `pull_request_target` is added later, this becomes a credential theft vector.

## Google OAuth client secret handling

```typescript
clientSecretValue: cdk.SecretValue.unsafePlainText(googleClientSecret),
```

This embeds the secret in plaintext in the synthesized CloudFormation template. Anyone with `cloudformation:GetTemplate` access or access to `cdk.out/` can read the Google client secret. Store it in Secrets Manager and reference via `SecretValue.secretsManager('secret-name')`.
